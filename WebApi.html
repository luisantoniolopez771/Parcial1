Unidad II: Tecnologías de Web APIs

Conceptos Fundamentales

Las Web APIs son interfaces que permiten el intercambio de datos entre un cliente y un servidor mediante el protocolo HTTP. Facilitan la integración de servicios externos como Maps, PayPal o redes sociales en nuestras aplicaciones.

Arquitectura y Funcionamiento

Tipos y Arquitecturas

  • Por visibilidad: Existen APIs públicas (abiertas), privadas (internas) y de socios.
  • Arquitectura REST: La más común actualmente, basada en recursos y el uso de métodos HTTP (GET, POST, PUT, DELETE).
  • SOAP: Un protocolo más rígido basado en XML, utilizado frecuentemente en entornos financieros.

Protocolos y Seguridad

  • HTTP/HTTPS: El protocolo base para la transferencia de datos en la web.
  • JSON: El formato de intercambio de datos más ligero y utilizado por las Web APIs modernas.
  • JSON Web Token (JWT): Un estándar abierto para transmitir información de forma segura entre las partes como un objeto JSON. Se usa principalmente para la autenticación y el intercambio de información confiable mediante firmas digitales.

HTTP

El protocolo HTTP (Hypertext Transfer Protocol) es el cimiento fundamental sobre el cual operan las Web APIs modernas, actuando como el lenguaje universal que permite la transferencia de datos entre un cliente (como un navegador web o una aplicación móvil) y un servidor. Su funcionamiento se basa en un ciclo continuo de "petición y respuesta" (request-response). Cuando un cliente necesita interactuar con la API, envía una petición estructurada que utiliza verbos o métodos HTTP para definir la intención de la acción: GET para consultar información, POST para crear nuevos registros, PUT o PATCH para actualizar datos existentes, y DELETE para eliminarlos.

SOAP

A diferencia de las aproximaciones más modernas y ligeras, SOAP es un protocolo estándar oficial y altamente estructurado, mantenido por el W3C, que fue diseñado para el intercambio formal de información en servicios web. Su característica más distintiva es que utiliza de forma exclusiva XML (eXtensible Markup Language) para el formato de todos sus mensajes. Esta estandarización requiere que cada petición y respuesta viaje dentro de un "sobre" lógico (Envelope), el cual se divide obligatoriamente en un encabezado (Header) para metadatos de seguridad o enrutamiento, un cuerpo (Body) que contiene los datos reales de la operación, y un bloque de fallos (Fault) dedicado exclusivamente al manejo detallado de errores.

REST

A diferencia de SOAP, REST no es un protocolo estricto, sino un estilo arquitectónico introducido por Roy Fielding en el año 2000. Hoy en día, se ha convertido en el estándar de la industria para el desarrollo de Web APIs debido a su simplicidad y alta eficiencia. En lugar de crear reglas complejas y sobres XML pesados, REST aprovecha al máximo la infraestructura existente de la web, utilizando directamente los métodos estándar del protocolo HTTP (GET, POST, PUT, DELETE) para realizar operaciones CRUD (Crear, Leer, Actualizar, Eliminar) sobre recursos. Estos recursos (ya sean usuarios, productos o facturas) se exponen mediante URLs lógicas, limpias y estructuradas, como por ejemplo: api/v1/productos/45.

Jason Web Token

JSON Web Token (estándar RFC 7519) es un método seguro y compacto para transmitir información entre distintas partes mediante un objeto JSON. En el contexto moderno del desarrollo de Web APIs, especialmente bajo la arquitectura REST, JWT se ha consolidado como el estándar principal para la autenticación y autorización. A diferencia de los métodos tradicionales que dependen de almacenar identificadores de sesión en la memoria del servidor o en una base de datos, JWT es completamente autónomo (self-contained). Esto significa que el propio token lleva consigo toda la información necesaria sobre el usuario (conocida como claims o afirmaciones), como su ID, sus roles de acceso y el tiempo de expiración de la credencial.
